26 /*
27  Implementation notes:
28 
29  - An OOP style is used where each "class" is made up of a struct and a number of functions prefixed with the struct name.
30 
31  - struct fields that are const are readonly. Either they are set in a create function and can never be changed, or they can only
32  be changed by calling a function.
33 
34  - Inheritance is done using a struct field named "super" as the first field, allowing the struct to be cast to its "super class".
35  This works because a pointer to a struct is guaranteed to be a pointer to the first struct field.
36 
37  - Classes intended for inheritance provide init/deinit functions which subclasses must call in their create/dispose functions.
38 
39  - Polymorphism is done by a base class providing function pointers in its init function. The public API delegates to this
40  function.
41 
42  - Subclasses do not provide a dispose function, instead the base class' dispose function should be used, which will delegate to
43  a dispose function.
44 
45  - Classes not designed for inheritance cannot be extended. They may use an internal subclass to hide private data and don't
46  expose function pointers.
47 
48  - The public API hides implementation details such init/deinit functions. An internal API is exposed in extension.h to allow
49  classes to be extended. Internal functions begin with underscore (_).
50 
51  - OOP in C tends to lose type safety. Macros are provided in extension.h to give context for why a cast is being done.
52  */
